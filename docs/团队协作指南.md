## 二、各角色工作方式

### Oxy：专注Unity可视化操作

**做**：
- Unity场景搭建、UI布局、Prefab制作
- 美术/音效资源设置

**不做**：
- 大量代码（让AI生成）
- 数据配置（让阿喵做）
- 复杂算法（让好捅天破做）

### 阿喵：用AI实现数据配置和简单系统

**让AI生成代码**：
```
"帮我创建 [功能名]：
需求：1. xxx 2. xxx
技术要求：使用EventCenter、TimerMgr
代码路径：Assets/Scripts/Game/[模块]/
请给出完整代码"
```

**遇到问题**：
1. 代码报错 → 把错误发给AI
2. 功能不对 → 描述期望vs实际
3. AI解决不了 → 问好捅天破

**成长路径**：
- Day 1：简单数据类
- Day 2-3：独立系统
- Day 4+：复杂交互

### 好捅天破：核心架构并兜底

**用AI快速实现**：
```
"帮我设计游戏流程控制器：
1. 管理关卡加载、暂停、胜负判定
2. 使用EventCenter、UIMgr
3. 考虑多人协作解耦
4. 给出完整代码"
```

**兜底时机**：
- 阿喵卡住超过30分钟
- Oxy的UI对接出问题
- 架构设计需要调整
- 发现重大bug

## 三、Git工作流

### 分支结构

```
develop (主开发分支)
├── develop_Oxy
├── develop_tianpo
└── develop_阿喵
```

### 日常操作

```bash
# 每天开始
git checkout develop_[你的名字]
git pull origin develop

# 开发中
git add .
git commit -m "[模块] 简短描述"

# 完成功能后
git push origin develop_[你的名字]

# 合并到develop（通知团队后再做）
git checkout develop
git pull origin develop
git merge develop_[你的名字]
git push origin develop
```

### 提交规范

```
[Card] 实现卡牌拖拽
[AI] 老师巡逻状态机
[Data] 创建关卡配置
[UI] 游戏主界面布局
[Fix] 修复读条bug
```

### 冲突预防

**规则1：按模块隔离**
- 每个人只修改自己的目录

**规则2：合并前通知**
- 在群里告知要合并了

**规则3：使用事件中心通信**
- 模块间通过EventCenter通信，避免直接引用
